# 代理节点全死时，浏览器设了代理也可能打不开——切 TUN 模式
type: diagnosis
date: 2026-06-22
source: Vortex/SakuraCat 订阅过期，112 个节点全死，浏览器无法访问任何外网

## 现象
- 所有代理节点离线（订阅过期），Clash 规则中 `MATCH,节点选择` 全部不可达
- 浏览器设置了系统代理 `127.0.0.1:7897`
- DIRECT 规则匹配的站点（`.cn`、`.com.cn`）在浏览器打不开
- 但 `curl --proxy 127.0.0.1:7897` 能通（curl → 代理 → DIRECT 规则生效 → 直连成功）

## 根因
系统代理不等于 TUN。代理节点全死时：

1. 浏览器把请求交给 `127.0.0.1:7897` → Clash 进程
2. Clash 按规则匹配，直连域名走 DIRECT
3. 但浏览器到 Clash 之间的连接链可能因各种原因断掉（进程状态异常、端口处理队列满、超时重试策略差异等）
4. curl 每次新连接，不受浏览器长连接/连接池/超时策略影响，所以能通

本质：**应用层代理（HTTP/SOCKS5）依赖代理进程健康状态，节点全死是进程的异常状态，不能假设它处理 DIRECT 也正常。**

## 修复
**切 TUN 模式。** TUN 是网络层（虚拟网卡），不经过代理进程的应用层处理，DIRECT 流量直接从网卡出去。SakuraCat TUN 模式解决。

不推荐的做法（这次踩坑了）：
- 搜替代机场网站 — 找到也是假站
- 改 Clash 规则加 DOMAIN-SUFFIX — 规则已经覆盖了，不是规则问题
- 反复测 curl 验证 — curl 能通不代表浏览器能通

## 预防
- 节点全死时，第一条排查不是"哪个站能访问"，而是"切 TUN 模式"
- curl 能通 ≠ 浏览器能通，尤其在代理进程异常状态下。不要用 curl 成功来排除代理问题
- 订阅过期前 7 天续费，避免掉进"没网→没法续费"的死循环
