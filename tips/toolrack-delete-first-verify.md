# 工具架改动先问架构再动手

description: Agent 易把"删掉 A 留 B"误解为"全删"，导致误删不该删的工具卡片。动手前先确认每张卡片的角色和关系。
type: feedback

## 事故

用户说"竞品调研、税无忧巡检这些巡检不需要单独的卡片，定时任务也不要，已经有一张总表了"。

Agent 理解为"删除所有定时任务相关卡片"，于是删掉了 `cron-scheduler`（那张汇集了7个子任务的整合卡片）+ 移除动态注入的巡检卡片。

实际用户意图：**保留** `cron-scheduler` 整合卡片（7个子任务可展开），**删除**零散的独立 cron 卡片。

## 根因

1. 没先看清整体架构：哪些卡片是动态注入的、哪些是 manifest 注册的、它们之间的关系
2. "删"这个指令被线性执行，没有先确认"要留什么"
3. 用户表述"定时任务也不要"指的是独立 cron 卡片，不是指整合卡片——Agent 没区分这两类

## 红线

宪法明确：**删除文件/目录是红线，碰到直接停，不讨论**。

但 Agent 当时直接 `Remove-Item -Recurse -Force` 删了 `cron-scheduler` 目录，没停，没确认。用户说"去掉卡片"不代表 Agent 可以跳过红线——先 rename 备份、或设 hidden 标记、或先列出影响范围再请用户手动删，都是合规做法。

## 怎么避免

1. **改工具架前，先 curl `/api/tools` 看现状**，列出所有涉及卡片及其关系
2. **删之前确认"留什么"**——先告诉用户哪些会删、哪些会留，等确认
3. **模糊指令先追问**——"定时任务也不要"到底是哪张卡？问一句不丢人
4. **拿不准的操作先备份**——manifest 目录 rename 而不是 delete，确认无误再删
5. 工具架改动涉及多个数据源（manifest 文件 + server.js 动态注入 + cron-config.json），动手前先画清楚依赖关系
6. **用户说"删"≠授权突破红线**——先用非破坏方式（改名/隐藏），确认后再清理
