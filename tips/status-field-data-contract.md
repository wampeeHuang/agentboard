# 状态字段不只是枚举标签——它隐含数据完整性契约
type: diagnosis
date: 2026-07-06
source: 猫波信号站磁盘-飞书同步，2条孤儿成品入库标"待发布"但缺评分/URL/摘要

## 现象

1. 磁盘有 2 个已生产完毕的成品文件夹（视频+封面+面板+字幕完整）
2. 创建飞书记录时只设了 Slug/标题/状态=待发布/频道/嘉宾
3. 评分字段（时效性/独占性/人物权威/长期价值/总分）全空，URL 空，中文摘要空
4. 用户指出"待发布不就已经生产完毕了吗"——才发现状态标签隐含了字段完整性要求

## 根因

Agent 把"设置状态"理解为机械操作——改一个枚举字段的值。但实际上：

- `待发布` 是一个**数据完整性断言**：它声称"这条记录已经走完候选→生产的全流程"
- 既然走完了全流程，以下字段必然有值：URL、评分四维、总分、中文摘要、YouTube 播放量/点赞
- 只设状态不补依赖字段 = 创建了语义矛盾的记录（状态说"生产完毕"，但 URL 为空意味着"没下载过视频"）

**这不是飞书特有的问题。** 任何状态机系统里，状态流转都隐含字段完整性约束：订单"已发货"必须有物流单号、PR "Ready for Review" 必须有 diff、部署"Live" 必须有版本号。

## 修复

1. 对照已有"待发布"记录，补全所有字段（URL/评分/摘要/YouTube 数据）
2. 飞书公式字段（总分）自动重算，无需手动设

## 预防

给任何记录设状态前，先做两件事：
1. **读一条同状态的已有记录**，看它有哪些字段、字段值长什么样
2. **问自己**：这个状态断言了什么？如果状态说"X 已完成"，那么完成 X 必然产生了哪些数据？那些数据都有了吗？

这条规则适用于飞书、GitHub Issues、数据库、任何有状态机的系统。
