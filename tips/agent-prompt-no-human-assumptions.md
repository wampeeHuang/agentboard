# Agent prompt 禁用人类交互假设

type: principle
date: 2026-06-28
source: 猫波信号站 cron prompt 审计 — "1-2% 黄灯警告需人工确认" 对自动化 agent 无意义

## 现象
- cron agent 读取包含"人工判断""手动操作""需人工确认"的 prompt 后，行为不确定
- Agent 不知道这些操作应该由谁执行——是跳过、报错、还是等待人类输入
- 审计发现 20 处此类问题，6 处直接面向 cron agent，最严重的会导致 agent 在评分阶段停顿等待不存在的"人工"

## 根因
LLM agent 把 prompt 中的人类交互指令视为执行阻塞点。写 prompt 的人假设读者是人类操作者，但 cron agent 是自动化程序——它不会"去问一下同事"，也不会"人工判断后填入"。

这是写作者视角错误：用 SOP 文档的语体写 agent 指令。

## 修复
自动化 agent 的 prompt 三条铁律：

1. **无灰色地带** — 所有决策分支必须闭合到确定动作。禁止"视情况而定""人工判断""maybe"。每个条件分支的出口必须是具体操作
2. **无人类指代** — 禁止"手动""人工""找同事确认""问一下"。agent 应该执行的动作直接写动作，真需要人类的动作用「输出提醒：需人类完成 X」显式标注
3. **不假设执行环境** — agent 不知道自己是在 cron 里跑、在对话里跑、还是在 CI 里跑。prompt 不依赖执行上下文

## 预防
- 写任何会进 cron job 的 prompt/文档前，自问：「一个没有眼睛没有手的程序读到这句话，知道该做什么吗？」
- 代码审查门禁：grep `人工|手动|确认|判断|视情况|maybe|perhaps` → 命中即拒绝
- 区分两类文档：人读的 SOP（可以有"人工判断"）和 agent 读的 prompt（不能有）。不要用同一种语体写两种文档
