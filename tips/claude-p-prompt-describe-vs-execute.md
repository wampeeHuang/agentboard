# Claude -p 模式下 agent 描述 prompt 而非执行

type: diagnosis
date: 2026-06-26
source: 进化猫日报×2 连续 3 天 output_missing，stdout 显示 agent 在总结模板结构

## 现象
- `claude -p` 触发的 cron 任务，exit code 0（成功），但实际产出文件未创建
- stdout 内容为 agent 对 prompt 的分析性描述：「这是…的 prompt 模板。结构清晰：Gate 2…需要我做什么？是跑一次还是修改这个 prompt？」
- scheduler 侧 `output_missing`，agent 侧无 stderr

## 根因
Agent 行为受 prompt 开头的框架效应影响。evopearl gate-common.md 模板以「## 门禁架构（Gate Architecture）」说明文开头，agent 将整段解读为「待审阅的文档」而非「待执行的任务」。`-p` 模式单次 turn 即退出，agent 没有第二次机会纠正。

这是 LLM agent 的系统性问题——prompt 开头的语体决定执行/分析模式，与内容是否包含执行指令无关。

## 修复
runner.js 在 prompt 拼接阶段强制注入执行前缀，不依赖模板写法：

```
var prompt = '立即执行以下任务。不询问、不确认、不总结模板结构。直接开始工作。\n\n' + task.prompt;
```

runner 管「做不做」，模板只管「做什么」。

## 预防
- 任何通过 `claude -p` 触发的自动化任务，prompt 第一位必须是执行指令，不能是说明文
- 产出验证 `output_missing` + stdout 像「分析报告」→ 大概率是此模式
- 不要只修模板——修 runner，让所有模板天然免疫

## 附：AI 修 bug 时的补丁倾向（元认知）

同一个事件暴露了 AI agent 在修 bug 时的系统性偏差：

**现象**：用户说「异常的也帮我修」，AI 第一方案是改 `gate-common.md` 模板——在模板顶部加一行执行指令。两个任务能好，下次另一个人写新模板说明文开头，一样复发。

**根因**：AI 的默认推理路径是「哪里坏了修哪里」。看到 agent 输出不对劲 → 改发给 agent 的内容。不会自动追问「为什么这里需要靠运气才能正确执行」。这是联想的短路——最近因（prompt内容）掩盖了远因（没有强制执行机制的架构）。

**为什么用户能发现而 AI 不能**：用户问的是「这是打补丁还是架构上的调整啊？」——关心的是方案的分层归属。AI 关心的是「怎么让这个东西工作」。视角差：用户在架构层思考，AI 在实例层思考。

**纠正方法**：修任何 bug 前先问自己——「这个修法是只修了当前实例，还是修了整类问题？」 如果只修了实例，往上翻一层找通用的注入点。三个检验：
1. 换了另一个任务/模板，同样的故障会复发吗？
2. 有没有一个地方改了之后，所有同类场景自动免疫？
3. 这个修复是在管「做什么」还是管「做不做」？前者是补丁，后者是架构。
